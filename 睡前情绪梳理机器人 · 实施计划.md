# 睡前情绪梳理机器人 · 实施计划

## Context（为什么做这件事）

课设项目（树莓派离线语音陪伴机器人，fork 自 be-more-agent），下周要交

核心决定：把产品从"泛泛的陪伴机器人"**收窄为"睡前梳理、调节情绪的聊天机器人"**。 所有功能取舍服从一条单向主线：**通电开机 → 说出今天 → 情绪落地 → 放松音频 → 睡着 → 自动关机**。 成功指标是"用户睡着"，不是"用得久"。

当前代码 [agent.py](vscode-webview://0hp3ngd3719b7a1hus2fq579vlbf3jq6j0v1412gac0hei3tgdsv/agent.py) 是 ~1200 行单文件 `BotGUI` 类，主循环是**开放式问答** （[agent.py:542](vscode-webview://0hp3ngd3719b7a1hus2fq579vlbf3jq6j0v1412gac0hei3tgdsv/agent.py#L542)），带着与睡前场景无关的包袱：摄像头/视觉模型、联网搜索、 tkinter 全屏 GUI、英文游戏音效。目标流程是一条**单向状态机**，与现有开放循环是两套骨架。

## Part A · 首轮延迟（原"冷启动"，已重新定性）

> 用户澄清：不是程序启动慢，而是程序已开好、首句处理明显慢于第二句。 实测数据：STT whisper **2.4s** / **LLM 首 Token 16.12s** / TTS piper **3s**。瓶颈 = LLM 首 Token。

**根因**：[agent.py:578](vscode-webview://0hp3ngd3719b7a1hus2fq579vlbf3jq6j0v1412gac0hei3tgdsv/agent.py#L578) `warm_up_logic` 用空 prompt 预热，只载入权重、 **未评估 system prompt 前缀**。第1轮真聊天要对整段前缀做 prompt-eval（16s）；第2轮 Ollama 复用 KV-cache 前缀 → 快。"第2轮快"已实证缓存有效。

**修复（高性价比，约 5 行，零功能损失）**：

- 改 `warm_up_logic`：把空 prompt 预热换成跑一次真实 `ollama.chat`， `messages = self.permanent_memory + [{"role":"user","content":"你好"}]`，`stream=False`，丢弃输出， **不**写入 memory。这样第1轮要用的 KV 前缀提前建好，真实第1轮退化成"第2轮"速度。
- 验证用代码里现成的计时器：[agent.py:874](vscode-webview://0hp3ngd3719b7a1hus2fq579vlbf3jq6j0v1412gac0hei3tgdsv/agent.py#L874) `LLM 首Token延迟` 对比修复前后第1轮。

**明确不做**（低性价比）：STT/TTS 优化、换模型、改 `num_thread`。 **叠加红利**（属于档2/档3，不额外花成本）：窄 prompt + 砍历史 → 前缀更短 → 首 Token 更快。
